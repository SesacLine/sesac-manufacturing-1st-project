> 대상: `manufacturing_agent/` 패키지의 멀티턴 대화 유지와 컨텍스트 관리 로직 전반

## 1. 큰 그림 — 컨텍스트는 두 개의 레이어로 유지된다

이 에이전트의 멀티턴은 **서로 다른 두 종류의 영속 저장소**가 협력해서 만들어진다. 이 이중 구조를 이해하는 것이 전체 분석의 출발점이다.

| 레이어 | 구현 | 저장 단위 | 역할 |
|--------|------|-----------|------|
| **(A) Graph Checkpointer** | `SqliteSaver` + `JsonPlusSerializer` (`graph/build.py:75-80`) | `ManufacturingState` 전체 스냅샷 (thread_id 기준) | 그래프 실행 중단/재개, `messages` 누적, 직전 턴의 typed artifact 복원 |
| **(B) ConversationStore (장기 메모리)** | 직접 만든 SQLite 래퍼 (`memory/store.py`) | `turns`, `summaries`, `diagnosis_contexts`, `context_state` | 대화 턴 로그, 요약, **재사용 가능한 진단 스냅샷(DiagnosisContext)** |

```python
# runtime.py:8-12 — 두 레이어 중 (A)의 진입점
sql_saver = make_sqlite_saver(CHECKPOINT_DB)          # 레이어 A
app = build_graph(checkpointer=sql_saver)
```

```python
# memory/store.py:251 — 레이어 B는 모듈 전역 싱글턴
conversation_store = ConversationStore()
```

핵심은 **같은 thread_id 가 두 저장소를 잇는 키**라는 점이다. `runtime.make_runnable_config` 가 `configurable.thread_id` 를 넘기면 (A)는 그 thread의 체크포인트를 복원하고, (B)는 `WHERE thread_id=?` 로 같은 대화의 턴/요약/진단 스냅샷을 조회한다.

```python
# runtime.py:25 — thread_id/user_id가 멀티턴 식별 키
configurable = {"thread_id": thread_id, "user_id": user_id}
```

## 2. 한 턴의 컨텍스트 처리 파이프라인

그래프 경로(`graph/build.py:45-63`)에서 컨텍스트와 직접 관련된 노드는 다음과 같다.

```
START → intake_gate → context_manager → supervisor_planner → orchestrator_dispatcher
        → (prediction/evidence/sql agent ↔ gate 루프) → final_answer → output_safety_gate
        → memory_writer → END
```

- **`context_manager`** : 입력 시점에 "과거를 어떻게 가져올지" 결정 (읽기 + 해석)
- **`memory_writer`** : 종료 시점에 "이번 턴을 어떻게 남길지" 결정 (쓰기)

이 두 노드가 멀티턴 컨텍스트의 알파이자 오메가다.

### 2.1 `context_manager` — 읽기·해석 (`context/manager.py:24-77`)

한 턴이 시작될 때 수행하는 6단계:

```python
# context/manager.py:41-55 (요약)
selected = select_context(msg, user_id, conversation_store, structured, thread_id=thread_id)  # ① 후보 수집
selected["previous_*_summary"] = ...                                                            # ② 직전 artifact 요약 주입
checkpoint_turns = _messages_to_recent_turns(state.get("messages", []), ...)                    # ③ 체크포인트 messages → 턴
selected["recent_turns"] = _dedup_turns(... + checkpoint_turns)[-RECENT_TURN_WINDOW:]           # ④ store턴+checkpoint턴 병합·중복제거
selected["context_carryover"] = _llm_context_carryover(msg, selected)                           # ⑤ LLM: 후속질문 판정
selected["context_resolution"] = resolve_context(msg, selected)                                 # ⑥ LLM: 입력 snapshot 재사용 판정
merged, warnings = normalize_context(selected)
packet, agent_ctx = pack_contexts(msg, merged, selected, warnings)
```

주목할 점:

- **③+④ 이중 소스 병합**: 최근 대화 턴을 (B)store의 `recent_turns` 와 (A)checkpoint의 `messages` 두 곳에서 가져온 뒤 `(role, content)` 키로 중복 제거한다(`context/packer.py:30-41`). 두 저장소가 같은 대화를 이중 노출하는 문제를 코드로 막아둔 점은 좋다.
- **체크포인트 dict 보정**: 복원 시 typed artifact가 dict로 올라올 수 있어 `model_validate` 로 되살린다(`manager.py:37-39`). 직렬화 round-trip을 의식한 방어.

### 2.2 `select_context` — "자동 보완 금지" 정책 (`context/selector.py:9-33`)

```python
# context/selector.py:16-23
current_vals = {**nl_vals, **structured}          # 자연어 추출값 + 구조화 입력
recent = store.recent_turns(...)
clean_recent = [t for t in recent if not detect_injection(t["content"])]   # injection 턴 제거
active_context = store.get_active_context(...)     # 직전 진단 snapshot 1개
recent_contexts = store.get_recent_contexts(..., limit=5)   # 최근 진단 snapshot 후보
```

여기서 설계 의도가 분명하다 — **이전 feature를 자동으로 끌어와 채우지 않는다**. `policy.py:32-41`의 CONTEXT_RULES 3·4번("현재 입력값이 우선, 현재값 없는 feature만 보완")과 selector docstring이 일관되게 "feature별 최신값 자동 fetch 금지"를 명시한다. 이는 제조 진단에서 **오래된 센서값을 슬쩍 섞어 잘못된 진단을 내는 것**을 막는 안전 지향 결정이다.

### 2.3 컨텍스트 해석을 LLM 2단계로 분리

이 시스템의 가장 특징적인 부분이다. "regex 키워드 매칭"을 버리고 **두 개의 독립된 LLM 호출**로 의미 기반 해석을 한다.

**⑤ Carryover (`packer.py:60-108`)** — "이번 발화가 이전 artifact(prediction/sql/evidence)를 참조하나?"

```python
# packer.py:90-105 — LLM 출력 후 계약 정합성 보정
if decision.is_followup and not decision.referenced_artifacts:
    # uses_previous_* 플래그로부터 referenced_artifacts 역추론
if not decision.is_followup:
    # 후속질문이 아니면 모든 참조 플래그 강제 False
```

**⑥ Resolution (`packer.py:111-255`)** — "이전 진단 입력 snapshot을 어떻게 쓰나?" 5개 모드:

| mode | 의미 |
|------|------|
| `CURRENT_ONLY` | 이번 턴에 직접 말한 값만 사용 (기본) |
| `USE_ACTIVE` | "방금 그 조건 그대로" → active snapshot 전체 재사용 |
| `PATCH_ACTIVE` | "토크만 70으로 바꿔서" → active에 변경값만 덮어씀 |
| `SELECT_HISTORY` | 과거 특정 조건 1개 지칭 |
| `REFER_ACTIVE_RESULT` | 재진단 아님, 직전 "결과"만 참조 |

`resolve_context` 는 LLM이 고른 mode를 **코드가 결정적으로 검증**한다:

```python
# packer.py:209-211 — base 없으면 안전하게 강등
if mode in {"USE_ACTIVE", "PATCH_ACTIVE"} and not base:
    warnings.append("재사용할 active 진단 context가 없어 현재 입력만 사용합니다.")
    mode = "CURRENT_ONLY"
```

```python
# packer.py:146-148 — patch는 "이번 턴 현재값"에 있는 키만 허용(과거값 임의 변경 차단)
def _filter_patch_values(values, current_values):
    allowed = set(current_values)
    return {k: v for k, v in values.items() if k in allowed}
```

**LLM의 판단을 그대로 믿지 않고, mode별 전제(base 존재 여부, patch 키 화이트리스트)를 코드로 강제 검증한 뒤 resolved_features를 만든다**는 점이 이 설계의 핵심 강점이다.

### 2.4 정규화 — 값마다 출처(provenance)를 태깅 (`context/normalizer.py:11-42`)

```python
# normalizer.py:31-37 — feature 하나하나에 source/is_current 부여
is_current = name in current_keys and (mode == "CURRENT_ONLY" or name in changed)
if is_current:        source = "current"
elif name in reused:  source = "active_context" if mode in {USE_ACTIVE,PATCH_ACTIVE} else "history_context"
else:                 source = "context"
```

진단에 들어간 각 값이 "이번 턴 값인지 / 재사용 값인지"를 `MachineValue.source`, `is_current` 로 추적한다. 최종 답변에서 "변경 입력 / 재사용한 진단 context feature"를 구분해 보여줄 수 있는 근거가 된다(`final_answer_node.py:224-228`).

### 2.5 패킹 — 노드별로 필요한 만큼만 잘라 전달 (`packer.py:257-321`)

`pack_contexts` 는 전체 컨텍스트를 한 덩어리로 주지 않고, **agent별 `AgentContextPacket`** 으로 분리한다.

```python
# packer.py:302-320 (발췌) — prediction은 feature만, sql은 요약+failure_history_only 같은 식
"prediction_agent": ...selected_context={"features": feats, "missing": missing, "sources": ..., **context_meta}
"evidence_agent":   ...prior_results=prior_results   # 이전 요약 carryover 플래그 동반
"sql_agent":        ...selected_context={"failure_history_only": True, ...}
```

이는 `policy.py:32`의 CONTEXT_RULES 2번 "전체 이전 대화를 Agent에게 그대로 전달하지 않는다"를 구현한 부분이다. **컨텍스트 최소 노출(least-context) 원칙**을 패킹 단계에서 강제한다.

### 2.6 새 턴 진입 시 runtime artifact 리셋 (`manager.py:57-77`)

```python
# manager.py:60-76 — 이전 요약은 packet으로 옮긴 뒤, 살아있던 결과는 모두 None으로
"prediction_result": None, "evidence_bundle": None, "sql_result": None,
"final_answer": None, "execution_plan": None, ... "retry_counts": {},
```

체크포인트로 복원된 직전 턴의 결과를 **요약 형태로만 packet에 옮기고, raw artifact는 비운다**. 멀티턴에서 "이전 결과가 이번 턴 plan/gate에 유령처럼 영향" 주는 것을 막는 좋은 패턴이다.

### 2.7 `memory_writer` — 쓰기 (`nodes/memory_writer_node.py:57-97`)

턴 종료 시:

```python
# memory_writer_node.py:64-91 (요약)
conversation_store.add_turn(user_id, "user", msg, thread_id)
conversation_store.add_turn(user_id, "assistant", fa.answer, thread_id)
# prediction OK/PARTIAL + summary면 summary 저장
if _should_save_diagnosis_context(...):     # 진단 snapshot 저장 (재사용 후보가 됨)
    diag = DiagnosisContext(id=f"diag-{uuid4}", features=..., failure_types=..., ...)
    conversation_store.save_diagnosis_context(...)   # 이 snapshot이 다음 턴 active_context가 됨
# evidence/sql summary 저장
return {"messages": [AIMessage(content=fa.answer)]}   # 체크포인트 messages에도 누적
```

`_should_save_diagnosis_context` 의 가드가 꼼꼼하다(`memory_writer_node.py:44-54`): 진단이 OK/PARTIAL이고, selected_machine_values가 있고, intake가 차단되지 않았고, injection이 아닐 때만 저장. **신뢰할 수 없는 입력으로 만든 진단이 재사용 후보가 되는 것**을 막는다.

저장된 snapshot은 `save_diagnosis_context` 에서 `context_state.active_context_id` 로 승격되고 5개만 유지된다(`store.py:128-135, 148-155`). → 다음 턴 `get_active_context` 가 이걸 읽어 `USE_ACTIVE`/`PATCH_ACTIVE` 재사용 루프를 완성한다.

## 2.8 [보강] 체크포인터 심층 — `messages`는 실제로 LLM에 "대화"로 전달되지 않는다

멀티턴을 이야기할 때 흔히 "체크포인터가 최근 N턴을 LLM에 그대로 넘긴다"고 오해하기 쉽다. 이 코드에서는 **그렇지 않다.** 체크포인터의 `messages`는 다음 두 가지 용도로만 쓰인다.

1. **그래프 재개(resumability)와 typed artifact 복원** — 중단 지점부터 다시 실행하고, 직전 턴의 `prediction_result` 등을 dict→pydantic으로 되살림(`manager.py:37-39`).
2. **최근 대화 턴 요약의 한 입력 소스** — `messages`를 `_messages_to_recent_turns`로 펼친 뒤 store 턴과 합쳐 **문자열 한 줄 요약**으로 압축(`manager.py:47-51`).

LLM에 들어갈 때는 항상 `recent_turns_summary`라는 **`"role:content | role:content"` 평탄화 문자열**로 바뀐다(`packer.py:55-58`). 즉 **체크포인터의 message 배열이 chat history로 모델에 직접 주입되는 경로는 없다.**

### `messages`에는 사실상 assistant 답변만 쌓인다

초기 state는 `messages: []`이고(`runtime.py:46`), 사용자 입력을 `HumanMessage`로 넣는 노드가 없다. 유일하게 `messages`를 추가하는 곳은 `memory_writer`다.

```python
# memory_writer_node.py:97 — 매 턴 AI 답변만 messages에 append
return {"messages": [AIMessage(content=fa.answer)]} if fa else {}
```

`MessagesState`의 `add_messages` 리듀서가 이를 누적하므로, **체크포인트 `messages`는 "AI 답변 히스토리"에 가깝다.** 사용자 발화 원문은 (B) ConversationStore의 `turns` 테이블에만 정확히 남는다(`memory_writer_node.py:64`). `context_manager`가 두 소스를 합쳐 dedup하는 이유가 여기 있다 — store는 user+assistant 전부, checkpoint는 assistant만 → assistant가 겹쳐서 중복 제거가 필요(`manager.py:51`, `packer.py:30-41`).

> **함의(아쉬운 점)**: turn 요약의 실질 출처는 store다. 체크포인트 `messages`는 turn 소스로는 거의 중복이고, 진짜 가치는 "재개 + artifact 복원"에 있다. 둘을 한 번에 묶어 "체크포인터가 대화를 기억한다"고 설명하면 부정확하다.

## 2.9 [보강] "최근 8턴 전부 전달" → 실제로는 50 윈도우 + role 기반 컷

예전 인상으로 남아있는 "limit 8턴 내 모든 대화 전달"은 **현재 코드에서 부분적으로만 맞다.**

- `ConversationStore.recent_turns(limit=8)`의 `8`은 **함수 기본값일 뿐**이고(`store.py:158`), 실제 호출은 전부 `RECENT_TURN_WINDOW = 50`으로 덮어쓴다(`selector.py:20`, `manager.py:47,51`, `intake_gate.py:178`). → **유효 윈도우는 8이 아니라 50턴.** 기본값 8은 사실상 사용되지 않는 잔재다.

- "조건에 따라 잘라서 보낸다"는 부분도 실재하지만, 그 기준은 **RAG 관련도가 아니라 발화 role**이다(`packer.py:43-58`).

```python
# packer.py:49-52 — user 턴은 전부 유지, assistant 답변만 최근 4개로 컷
if user_all:                                   # ← carryover/resolution/planner/sql 모두 user_all=True로 호출
    assistant_idx = [i for i, t in enumerate(seq) if t.get("role") == "assistant"]
    keep = set(assistant_idx[-assistant_limit:])     # assistant_limit = ASSISTANT_TURN_LIMIT = 4
    selected = [t for i, t in enumerate(seq) if t.get("role") != "assistant" or i in keep]
```

근거 주석(`packer.py:23-28`)이 명확하다: "사용자 질문은 의도 추적에 중요 → 전부 유지 / AI 답변은 길어 토큰을 많이 씀 → 최근 4개만." 즉 **컷의 동기는 토큰 비용(특히 긴 AI 답변)이지 RAG 품질 점수가 아니다.** RAG 관련도 기반 컷은 evidence_agent(문서 검색) 쪽 이야기이고, 여기 대화 요약 컷과는 별개다. (혼동했을 가능성이 큰 지점.)

정리하면 한 턴의 대화 요약은 **`min(50, 보유턴)` 윈도우에서 user 전부 + assistant 최근 4개**가 한 줄 문자열로 만들어져 LLM에 들어간다.

## 2.10 [보강] 컨텍스트 전달 채널 2종 — LangGraph(call_llm) vs PydanticAI(SQL)

이 시스템은 **성격이 다른 두 LLM 호출 경로**를 갖는다. 컨텍스트가 전달되는 방식이 서로 다르다.

### 채널 A — LangGraph 노드 내부 `call_llm`

intake / carryover / resolution / planner / replanner / final_answer가 여기 속한다.

- 모델: `gpt-4o`(default/final) 또는 `gpt-4o-mini`(classifier) (`config.py:92-97`).
- 호출 형태: `call_llm(system, user)`에서 **`user`는 손으로 만든 payload를 `json.dumps`한 문자열**이다. 대화는 항상 `recent_turns_summary` 한 줄로 평탄화돼 그 payload 안에 들어간다.

```python
# planner.py:108 — 대화/이전요약을 payload(JSON 문자열)로 직렬화해 넘긴다
raw = call_llm(SUPERVISOR_PLANNER_SYS, json.dumps(payload, ensure_ascii=False), tier="default")
```

- 재시도: `call_llm`의 지수 백오프(`config.py:130-163`). 출력: 자유 텍스트 → `_json_object`로 파싱.
- 사용량 계측: `record_llm_usage`로 OTel에 집계됨(`config.py:143-145`).

### 채널 B — PydanticAI Text-to-SQL 에이전트 (`sql_agent`)

SQL 생성만 담당하는 **완전히 분리된 에이전트**다.

```python
# evidence_agent.py:458-463 — 독립 모델·독립 스키마·독립 retry를 가진 PydanticAI Agent
agent = PydanticAIAgent(
    DEFAULT_PYDANTIC_SQL_MODEL,            # = "openai-chat:gpt-4.1-mini" (채널 A와 다른 모델!)
    output_type=SQLTextToSQLResponse,      # structured output (Pydantic)
    deps_type=SQLTextToSQLDeps,
    retries=2,                             # 자체 retry + output_validator의 ModelRetry
)
```

- **컨텍스트 주입 방식**: `ManufacturingState`도 `messages`도 받지 않는다. `_build_sql_context_summary`가 packet에서 필요한 조각만 뽑아 **별도 문자열**을 만들고(`evidence_agent.py:639-666`), `_text_to_sql_prompt`가 그것 + user_message + task_params를 **하나의 프롬프트 문자열**로 합쳐 넘긴다(`evidence_agent.py:510-518`).
- **PydanticAI의 대화 메모리(message_history) 기능은 쓰지 않는다.** 매 호출이 stateless이고, 멀티턴 맥락은 오직 위 `context_summary` 문자열을 통해서만 전달된다. 즉 여기서 PydanticAI는 "대화형 에이전트"가 아니라 **일회성 Text-to-SQL 함수**로 쓰인다.
- 실행 모델도 다르다: `await agent.run(...)`(async)를 `_run_coroutine_sync`(별도 스레드+`asyncio.run`)로 감싸 동기 LangGraph 노드에 끼워 넣는다(`evidence_agent.py:530-544`).
- **사용량 계측에서 빠진다**: `call_llm`을 거치지 않으므로 OTel 토큰 집계에 포함되지 않는다(`api/routers/usage.py:20`, `RUN.md:165`에 명시).

### 두 채널 비교

| 항목 | 채널 A (LangGraph `call_llm`) | 채널 B (PydanticAI SQL) |
|------|------------------------------|--------------------------|
| 컨텍스트 전달 | `recent_turns_summary` 포함 JSON payload **문자열** | `context_summary` **문자열** + user_message |
| message 배열 직접 주입 | ❌ (요약 문자열로 평탄화) | ❌ (요약 문자열로 평탄화) |
| 네이티브 대화 메모리 | LangGraph 체크포인터(state/messages) | **미사용** (호출마다 stateless) |
| 모델 | gpt-4o / gpt-4o-mini | gpt-4.1-mini (별도) |
| 재시도 | `call_llm` 백오프 | PydanticAI `retries=2` + `ModelRetry` |
| 출력 | 자유 텍스트 → 직접 JSON 파싱 | structured Pydantic + output_validator |
| 토큰 사용량 집계 | 포함 | **제외** |
| 실행 | 동기 | async → 동기 브리지 |

**결론(중요)**: 채널 A·B 모두 **대화를 결국 문자열로 평탄화**해서 넘긴다. 어느 쪽도 "chat message 배열"을 모델에 그대로 주지 않는다. 차이는 *어디서 그 문자열을 만드느냐*(A=context_manager가 만든 packet 요약, B=sql_agent가 packet에서 다시 추린 요약)와 *에이전트 런타임의 성격*(A=프롬프트 함수, B=구조화 출력 + 자체 retry를 갖춘 독립 에이전트지만 메모리는 끔)이다.

> **함의(아쉬운 점)**: ① 같은 멀티턴 맥락이 채널 A의 packet 요약과 채널 B의 `_build_sql_context_summary`에서 **두 번 따로 직렬화**된다 — 포맷이 어긋나면 SQL 에이전트가 보는 맥락과 planner가 보는 맥락이 미묘하게 달라질 수 있다. ② PydanticAI 호출이 **토큰 집계에서 빠져** 멀티턴 비용 모니터링에 사각지대가 생긴다. ③ PydanticAI의 message_history(네이티브 멀티턴) 대신 수동 문자열을 쓰므로, 향후 SQL 후속질문이 길어질 때 일관성·중복 부담을 떠안는다.

## 3. 멀티턴 시나리오로 본 동작

**시나리오: "토크 60, 공구마모 200으로 진단해줘" → "토크만 70으로 바꾸면?"**

1. 턴1: `CURRENT_ONLY` → 진단 → `memory_writer` 가 features `{torque:60, tool_wear:200, ...}` 를 DiagnosisContext로 저장 + active 승격.
2. 턴2: `context_manager` →
   - `select_context` 가 `active_context`(턴1 snapshot) 로드
   - `_llm_context_carryover`: is_followup=true
   - `resolve_context`: LLM이 `PATCH_ACTIVE` + `patch_values={torque:70}` 판정 → 코드가 base(active) 존재 확인, patch 키 화이트리스트 통과 → `resolved_features = {torque:70, tool_wear:200, ...}` (`packer.py:230-237`)
   - `normalize_context`: torque는 `source=current/changed`, 나머지는 `source=active_context/reused`
3. prediction_agent가 merged feature로 재진단, `final_answer` 가 "변경 입력: 토크 / 재사용한 진단 context feature: 공구마모…" 로 설명.

이 흐름이 **"부분 수정 후속 질문"을 자연어 키워드 없이 처리**하는 핵심 가치다.

## 4. 잘된 점 (강점)

1. **이중 저장소 역할 분리가 명확**. 체크포인터(실행 재개/messages)와 ConversationStore(의미 재사용 snapshot)를 섞지 않고, thread_id로만 연결한다.
2. **regex → LLM 2단계 의미 해석**. carryover(참조 여부)와 resolution(snapshot 재사용 방식)을 분리해 각각 단일 책임을 갖는다. 프롬프트도 "너는 task planner가 아니다"라고 역할 경계를 반복 강조(`packer.py:63, 112`).
3. **LLM 판단을 코드가 검증**. mode 강등(`packer.py:209-243`), patch 키 화이트리스트(`146-148`), carryover 플래그 정합성 보정(`90-105`) — LLM hallucination을 결정적 규칙으로 가둔다.
4. **provenance 추적**. 진단에 쓴 값마다 current/reuse/stale 출처를 달아 답변에서 투명하게 설명.
5. **안전 지향 컨텍스트 정책**. 자동 feature 보완 금지, injection 턴 필터(`selector.py:18`, `packer.py:14`), 신뢰 불가 입력의 snapshot 저장 차단(`memory_writer_node.py:44-54`).
6. **컨텍스트 최소 노출**. 노드별 AgentContextPacket으로 필요한 조각만 전달.
7. **thread 격리**. store의 모든 read가 `WHERE thread_id=?` 로 한정돼 다른 대화 누수를 막는다(`store.py:158-224`, 주석에도 명시).

## 5. 아쉬운 점 / 개선 포인트

### 5.1 [성능] 한 턴에 컨텍스트용 LLM 호출이 최소 2회 — 직렬·중복

`_llm_context_carryover`(⑤)와 `_llm_context_resolution`(⑥)이 **각각 별도 `call_llm`** 이고 `context_manager` 안에서 순차 실행된다(`manager.py:52-53`). 두 프롬프트는 동일한 `recent_turns_summary`/이전 요약을 입력으로 받는다(`packer.py:73-80` vs `151-160`).

```python
# packer.py:73, 156 — 같은 요약을 두 번 따로 계산해서 두 번 LLM에 보낸다
recent_summary = _summarize_recent_turns(selected.get("recent_turns") or [], user_all=True)  # carryover
... = _summarize_recent_turns(selected.get("recent_turns") or [], user_all=True)             # resolution
```

여기에 supervisor_planner도 또 LLM 호출이라, **답변 생성 전 분류용 LLM이 3회 직렬**. 후속질문이 아닐 때(대부분의 첫 턴)는 carryover가 사실상 불필요하다.
→ **개선**: (a) 두 판정을 1개 프롬프트/스키마로 합치거나, (b) `current_values`만 있고 `active_context`/요약이 전부 없으면 LLM 없이 `CURRENT_ONLY` 로 단락(short-circuit), (c) 두 호출을 병렬화.

### 5.2 [정확성] 최근 대화 윈도우의 토큰 폭주 가능성

`RECENT_TURN_WINDOW = 50`(`packer.py:27`)이고 `_summarize_recent_turns(..., user_all=True, chars=None)` 은 **사용자 턴을 전부, 원문 그대로(자르지 않고)** 이어붙인다(`packer.py:43-58`).

```python
# packer.py:49-52 — user 턴은 개수·길이 제한 없이 전부 보존
if user_all:
    selected = [t for ... if role != "assistant" or i in keep]   # assistant만 4개로 제한
```

긴 대화에서 사용자 발화가 길면 이 요약이 carryover/resolution/planner 프롬프트에 **그대로 3번 들어가** 토큰·비용·지연이 선형 증가한다. CONTEXT_RULES 9번("token budget 초과 시 우선순위")이 문서로만 존재하고 **실제 budget 집행 코드가 없다**.
→ **개선**: 사용자 턴도 문자수/토큰 캡 또는 오래된 턴 압축 요약(rolling summary) 도입.

### 5.3 [정확성] `RECENT_TURN_WINDOW` 의미가 store와 checkpoint에서 어긋남

`store.recent_turns(limit=50)` 는 최근 50턴을 가져오지만, checkpoint `messages` 는 전체 누적이라 `_messages_to_recent_turns(limit=50)` 도 50개. 합쳐서 dedup 후 다시 `[-50:]`(`manager.py:51`). 즉 **user/assistant가 섞인 50개**로 잘리는데, dedup 키가 `(role, content)` 라서 **동일 질문을 다른 턴에 반복하면 과거 맥락이 한 개로 접혀** 시간 순서가 왜곡될 수 있다(`packer.py:35-40`).
→ **개선**: dedup 키에 created_at/turn 인덱스 포함, 또는 checkpoint와 store 중 하나를 단일 출처로.

### 5.4 [정합성] `latest_summary` 가 thread 단위 "최신 1건"만 — 종류별 누적 손실

evidence/sql/prediction 요약은 `summaries` 테이블에 append되지만 재사용 시 `ORDER BY id DESC LIMIT 1` 로 **가장 최근 1건만** 읽는다(`store.py:189-200`). 멀티턴에서 "3턴 전 SQL 결과"를 참조하려 해도 그 사이 다른 SQL이 있었으면 덮인다. carryover가 `uses_previous_sql=true` 라 판단해도 정작 가져오는 건 직전 것뿐.
→ **개선**: turn_id 기반으로 특정 과거 artifact를 지목해 가져오는 경로(SELECT_HISTORY가 feature엔 있지만 sql/evidence 요약엔 없음).

### 5.5 [정확성] `is_stale` 가 항상 False — stale 정책이 死문자

CONTEXT_RULES 8번("오래된 센서값은 stale 표시")과 `MachineValue.is_stale`, `PredictionResult.used_stale_features` 필드가 다 있는데, 실제로 stale을 세팅하는 코드가 없다.

```python
# normalizer.py:7-8 — 항상 is_stale=False 하드코딩
def _machine_value_from_context(...):
    return MachineValue(..., is_stale=False)
```

재사용 snapshot이 며칠 전 것이어도 "최신값"처럼 답변에 들어간다. `DiagnosisContext.created_at` 이 있으므로 경과시간 기반 stale 판정이 가능한데 미사용.
→ **개선**: `created_at` 과 현재 시각 비교로 임계 초과 시 stale 표시 + 답변 경고.

### 5.6 [신뢰성] 컨텍스트 LLM 호출 실패 시 비대칭 폴백

`resolve_context` 는 LLM 실패 시 `CURRENT_ONLY` 로 안전 폴백한다(`packer.py:179-188`). 그런데 `_llm_context_carryover` 는 실패 시 `ContextCarryoverDecision()`(전부 False) 반환(`packer.py:107-108`) — 즉 **"후속질문 아님"으로 조용히 처리**. 사용자가 "방금 그거 더 설명해줘" 했는데 LLM이 일시 실패하면 맥락이 끊긴 채 일반 질문으로 처리되고, 사용자는 이유를 모른다.
→ **개선**: carryover 실패도 `context_warnings` 에 노출해 답변에 "이전 맥락 해석 실패" 신호 추가.

### 5.7 [확장성] ConversationStore 싱글턴 + 매 호출 connect

`conversation_store` 가 모듈 전역 싱글턴이고(`store.py:251`), 모든 read/write가 `with self._conn()` 으로 **매번 `sqlite3.connect`** 한다(`store.py:46-55`). 한 턴에 read만 6~7회(`select_context`). 노트북 데모엔 충분하나, FastAPI 동시요청(레포에 `[feat] build fastapi` 커밋 존재)에서는 connection-per-call + 단일 파일 SQLite가 병목·락 경합이 된다.
→ **개선**: connection pool 또는 thread-local, WAL 모드, 혹은 서버용 DB 분리.

### 5.8 [정확성] 진단 snapshot 가드와 carryover의 불일치

`memory_writer` 는 prediction이 OK/PARTIAL일 때만 DiagnosisContext를 저장한다(`memory_writer_node.py:69-71`). 따라서 **NEEDS_INPUT(입력 부족)으로 끝난 진단은 snapshot이 안 남는다**. 그런데 다음 턴에서 사용자가 "아까 그 입력에 토크만 추가"라고 하면 `USE_ACTIVE`/`PATCH_ACTIVE` 가 참조할 active가 없어 `CURRENT_ONLY` 로 강등되고, 사용자는 자기가 준 부분 입력이 사라졌다고 느낀다.
→ **개선**: 부분 입력도 `is_safe_to_reuse=False` 같은 별도 플래그로 저장해 patch 베이스로만 허용.

### 5.9 [유지보수] PredictionResult의 context 메타가 3중 중복

`context_mode`/`changed_features`/`reused_features` 가 `ContextResolution`, `AgentContextPacket.selected_context`, `PredictionResult` 세 곳에 복제된다(`prediction_agent.py:37-40` 이 셋을 일일이 fallback 체이닝). 단일 출처가 아니라 동기화 부담이 있다.
→ **개선**: resolution을 단일 출처로 두고 나머지는 참조만.

## 6. 한 줄 요약

> **"regex 대신 LLM 2단계 해석 + 코드 검증 + provenance 태깅 + 안전 가드"** 라는 방향성은 제조 진단 도메인에 잘 맞고 설계가 단정하다. 다만 **(1) 턴당 분류 LLM 다중 직렬 호출, (2) 토큰 budget 미집행, (3) stale 정책 미구현, (4) artifact 요약의 "최신 1건만" 한계** 가 멀티턴이 길어질수록 비용·정확도 양쪽에서 약점이 된다.
>
> 체크포인터 관점에서 한 가지 더: **체크포인터의 `messages`는 LLM에 "대화"로 직접 주입되지 않는다.** 멀티턴 맥락은 (A)LangGraph 채널과 (B)PydanticAI SQL 채널 **양쪽 모두에서 문자열 요약으로 평탄화**되어 전달되며, 유효 대화 윈도우는 (흔히 기억하는 8턴이 아니라) **50턴 + user 전부/assistant 최근 4개** 컷이다. PydanticAI 호출은 네이티브 대화 메모리를 끄고 수동 요약을 쓰며 **토큰 집계에서도 빠진다.**
